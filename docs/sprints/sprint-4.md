## Sprint 4: Google Forms統合によるリード獲得強化

### 実装する機能
- 診断結果に関わらず常にGoogle Forms表示: 全ての診断完了後にフォームを表示（問題有無で見出し・説明文を切り替え）
- フォーム埋め込みのレスポンシブ対応: PC/タブレット/スマホでの最適な表示
- 診断結果に応じた最適なメッセージング: 問題あり/なしで異なる訴求内容を自動切り替え

### スプリント契約（完了条件）
以下の全条件を満たした場合のみ、このスプリントは完了とする。

#### 問題判定ロジック
- [ ] SPF/DKIM/DMARCのいずれか1つでも未設定または設定エラーがある場合、「問題あり」と判定される
- [ ] 診断スコアが79点以下の場合も「問題あり」と判定される
- [ ] 問題判定時、診断結果セクション内に「has-issues」クラスが追加される
- [ ] 問題なしの場合、診断結果セクション内に「no-issues」クラスが追加される

#### Google Forms埋め込み
- [ ] 診断結果セクションの直後（CTAセクションの前）にフォームエリアが常に挿入される
- [ ] 問題ありの場合、見出しは「設定方法がわからない方、専門家に相談したい方へ」と表示される
- [ ] 問題なしの場合、見出しは「その他のご相談も承ります」と表示される
- [ ] Google FormsがiFrameで埋め込まれる（URL: https://docs.google.com/forms/d/e/1FAIpQLSecpCNK7VJye7qQSWmLCdOTQZiTdPm82K-MXKYCNAK_23ehbA/viewform?embedded=true）
- [ ] iFrameの高さは最低640px、幅は親要素の100%に設定される
- [ ] 問題ありの場合、「SPF/DKIM/DMARCの設定は専門的な知識が必要です。設定方法がわからない、または専門家に任せたいという方は、お気軽にご相談ください。」の説明文が表示される
- [ ] 問題なしの場合、「メール配信設定は問題ありません。その他、Webサイト保守・運用、AI業務効率化、集客改善などのご相談も承っております。」の説明文が表示される
- [ ] フォーム埋め込みエリアにはスムーズスクロールでアンカー移動できる

#### レスポンシブデザイン
- [ ] PC（1024px以上）: フォームが最大幅800pxで中央配置される
- [ ] タブレット（768px-1023px）: 左右に20pxのパディングを保ちつつ全幅表示される
- [ ] スマホ（767px以下）: 左右10pxのパディングで全幅表示される
- [ ] 全デバイスでフォームの文字が読みやすいサイズで表示される
- [ ] iFrameが親要素からはみ出さない

#### デザイン整合性
- [ ] フォームエリアの背景色が既存のCursor風デザインに合致している（ダークテーマ: #1a1a1a）
- [ ] 見出しのフォントサイズ・色が既存の他セクションと統一されている
- [ ] フォームエリアにソフトシャドウ（box-shadow）が適用されている
- [ ] フォームエリアの角が丸められている（border-radius: 12px）
- [ ] フォームエリアと他セクションとの間隔が適切である（margin: 2rem 0）

#### ユーザー体験
- [ ] 診断完了後、自動的にフォームエリアまでスムーズスクロールする（問題有無に関わらず）
- [ ] フォーム表示時にフェードインアニメーション（0.3秒）が適用される
- [ ] フォームの読み込み中は「フォームを読み込んでいます...」のメッセージが表示される
- [ ] フォーム読み込みエラー時は「フォームの読み込みに失敗しました。ページをリロードしてください」が表示される
- [ ] 診断結果が変わった時、既存のフォームを再利用し、見出しと説明文のみ更新される（重複表示しない）

#### アクセシビリティ
- [ ] フォームエリアに適切なARIAラベルが設定されている
- [ ] キーボードでフォームにフォーカスできる
- [ ] 問題ありの場合、スクリーンリーダーで「専門家への相談フォーム」と読み上げられる
- [ ] 問題なしの場合、スクリーンリーダーで「その他のご相談フォーム」と読み上げられる

#### トラッキング
- [ ] Google Analytics 4でフォーム表示イベントがトラッキングされる（event: form_displayed）
- [ ] フォームエリアへのスクロール到達がトラッキングされる（event: form_viewed）
- [ ] 診断結果の問題有無がカスタムディメンション「diagnosis_result」として記録される（値: 「has_issues」または「no_issues」）
- [ ] フォーム表示タイプがカスタムディメンション「form_type」として記録される（値: 「support_request」または「other_consultation」）

### 受入テストシナリオ

#### シナリオ1: 問題ありの診断
1. ドメイン「test-no-spf.example.com」を入力して診断
2. SPF未設定のエラーが表示される
3. 診断結果の直後にGoogle Formsが表示される
4. 見出しに「設定方法がわからない方、専門家に相談したい方へ」と表示される
5. 説明文に「SPF/DKIM/DMARCの設定は専門的な知識が必要です」と表示される
6. フォームまで自動スクロールされる
7. フォームに入力して送信できる

#### シナリオ2: 問題なしの診断
1. 全て設定済みのドメインを入力して診断
2. 診断スコアが80点以上表示される
3. 診断結果の直後にGoogle Formsが表示される
4. 見出しに「その他のご相談も承ります」と表示される
5. 説明文に「メール配信設定は問題ありません。その他、Webサイト保守・運用」と表示される
6. フォームまで自動スクロールされる
7. フォームに入力して送信できる

#### シナリオ3: レスポンシブ表示
1. Chrome DevToolsでモバイル表示に切り替える
2. 問題ありの診断を実行する
3. フォームが画面幅に収まって表示される
4. 横スクロールが発生しない
5. フォームの入力フィールドがタップしやすいサイズである
6. 問題なしの診断に切り替えて同様に確認する

#### シナリオ4: 連続診断での動的切り替え
1. 問題ありのドメインを診断する
2. 「設定方法がわからない方、専門家に相談したい方へ」の見出しを確認
3. 問題なしのドメインを再診断する
4. 見出しが「その他のご相談も承ります」に切り替わることを確認
5. 説明文も適切に切り替わることを確認
6. フォームが重複表示されず、1つだけ表示されることを確認
